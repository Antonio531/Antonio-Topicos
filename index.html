1.1 Conceptos de base de datos distribuidas

Un sistema de bases de datos distribuidas (DDBS) integra múltiples nodos interconectados para almacenar y procesar información de forma coordinada, presentando a los usuarios una vista lógica única aunque los datos estén repartidos físicamente en diferentes sitios. Sus objetivos principales son escalabilidad, disponibilidad y baja latencia.

Transparencias fundamentales

  • Fragmentación: oculta que una tabla está dividida en fragmentos.
  • Ubicación: evita que el usuario conozca en qué sitio vive cada fragmento.
  • Replicación: permite múltiples copias aparentando un solo dato.
  • Concurrencia y fallas: múltiples clientes y fallas sin comprometer la ilusión de unidad.

CAP y modelos de consistencia

En presencia de particiones de red, no se puede tener a la vez consistencia fuerte y disponibilidad total. Los sistemas eligen entre CP (consistencia fuerte) o AP (alta disponibilidad con consistencia eventual). También existen modelos intermedios (por quórum, monotónica, lectura de instantáneas).

Arquitecturas

  • Shared-nothing (la más usada para escalar): cada nodo administra CPU y almacenamiento propios.
  • Federada: integra motores heterogéneos bajo un catálogo global.
  • Primario-réplica / multi-líder: según el patrón de escritura.

1.2 Diseño de base de datos distribuidas

Diseñar implica decidir fragmentación, ubicación y replicación maximizando desempeño y resiliencia con el menor costo operativo.

Fragmentación

  • Horizontal (por rango o hash): reparte filas; favorece escalamiento de escritura.
  • Vertical: separa columnas; acerca atributos más usados.
  • Híbrida: combina ambas según el patrón de acceso.

Ubicación y replicación

  • Replicación síncrona: lecturas consistentes con mayor latencia.
  • Replicación asíncrona: mejor desempeño tolerando desfaces temporales.
  • Quórums R/W: garantiza intersección para visibilidad global.

Catálogo y enrutamiento

Un catálogo global mantiene metadatos de fragmentos, ubicaciones y reglas de enrutamiento. Se usan tablas de hash consistentes para minimizar movimientos cuando cambian los nodos.

Buenas prácticas: evitar puntos únicos de falla, balancear particiones, versionar el esquema global y registrar decisiones de particionado.

1.3 Procesamiento de operaciones de actualización distribuidas

El reto es que todas las réplicas converjan al mismo estado pese a fallas y concurrencia.

Patrones de escritura

  • Primario‑réplica: un nodo serializa escrituras por partición; sencillo pero dependiente del primario.
  • Multi‑líder: se puede escribir en varias réplicas; requiere resolución de conflictos.

Control de concurrencia y conflictos

  • Marcas de tiempo, relojes vectoriales, LWW y CRDTs para estados convergentes.
  • Idempotencia y reintentos con backoff para fallas intermitentes.

Atomicidad distribuida

  • Two‑Phase Commit (2PC): preparar → confirmar/abortar; simple pero puede bloquear.
  • Three‑Phase o sagas: reducen bloqueos o compensan pasos en procesos largos.

1.4 Procesamiento de consultas distribuidas

El optimizador descompone la consulta global en subconsultas locales y elige planes que minimizan transferencias y latencia.

Optimización de costos

  • Push‑down de selecciones/proyecciones al sitio origen.
  • Elegir si mover datos al operador o el operador a los datos.
  • Estrategias de unión: broadcast, partitioned hash join, semi‑join.

Reducción de tráfico

  • Filtros de Bloom para podar claves antes de mover filas.
  • Pipelining y paralelismo para superponer E/S y CPU.

Federación

En entornos heterogéneos, el coordinador traduce a dialectos locales y reensambla resultados respetando la semántica declarativa original.

1.5 Manejo de transacciones

Una transacción debe cumplir ACID y extenderlo a varios participantes conectados por una red no confiable.

Aislamiento

  • 2PL (bloqueo de dos fases), ordenamiento por marcas de tiempo y MVCC.
  • Lecturas de instantánea para alta concurrencia y garbage collection de versiones.

Interbloqueos y recuperación

  • Detección global, tiempos de vida o prevención con órdenes totales.
  • Logs WAL y checkpoints para rehacer/deshacer tras fallas.

Coordinación

  • 2PC/3PC para atomicidad; sagas para procesos de larga duración.

Reflexión personal

Aprendí que distribuir datos no es únicamente “partir tablas”; implica elegir compromisos explícitos entre consistencia, disponibilidad y latencia. Lo más retador fue interiorizar el efecto práctico de CAP y del 2PC sobre el tiempo de respuesta, pero al practicar con fragmentación por rango vs hash y con quórums de lectura/escritura entendí cómo se gobierna la convergencia. Me gustó el papel del optimizador empujando filtros para no mover bytes innecesarios y el valor de MVCC para lecturas rápidas; ahora veo que las decisiones de particionado y replicación que se toman al inicio determinan la simplicidad operativa y el rendimiento real del sistema que pienso construir.
